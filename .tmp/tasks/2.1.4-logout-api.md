## 概要
ユーザーのログアウト機能を実装するAPIエンドポイントの作成。セッションの削除、Cookieのクリア、Auth0からのログアウト（オプション）を含む。

## 実装内容

### 1. エンドポイント仕様
- **パス**: `/api/v1/auth/logout`
- **メソッド**: POST
- **認証**: 必須（有効なセッションIDが必要）
- **リクエストボディ**: 
  ```json
  {
    "redirect_to": "https://example.com/goodbye" // オプション：ログアウト後のリダイレクト先
  }
  ```
- **レスポンス**:
  - 成功時（200）:
    ```json
    {
      "success": true,
      "message": "Successfully logged out",
      "redirect_url": "https://example.com/goodbye" // redirect_toが指定された場合
    }
    ```
  - 認証エラー（401）:
    ```json
    {
      "success": false,
      "error": "unauthorized",
      "message": "No active session found"
    }
    ```

### 2. 処理フロー
1. **セッション検証**
   - CookieからセッションIDを取得
   - D1データベースでセッションの存在と有効性を確認
   - セッションが無効な場合は401エラー

2. **セッション削除**
   - D1データベースからセッションレコードを削除
   - 関連するすべてのセッションデータをクリーンアップ

3. **Cookie削除**
   - session_idクッキーを削除（maxAge: 0）
   - 他の認証関連クッキーがあれば削除

4. **Auth0ログアウト（オプション）**
   - グローバルログアウトが必要な場合、Auth0のログアウトURLを構築
   - returnToパラメータでリダイレクト先を指定

### 3. セキュリティ考慮事項
- **CSRF対策**: POSTメソッドを使用（GETは使用しない）
- **リダイレクト検証**: redirect_toパラメータのホワイトリスト検証
- **タイミング攻撃対策**: セッションの存在に関わらず一定時間で応答
- **ログ記録**: ログアウトイベントを記録（監査目的）

## メモ
- POSTメソッドを使用することでCSRF攻撃を防ぐ
- グローバルログアウト（Auth0側も含む）はオプションとして実装
- ログアウト後のリダイレクト先は検証が必要（オープンリダイレクト脆弱性対策）
- セッション削除は即座に反映される必要がある（キャッシュなし）
