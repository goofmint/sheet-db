# [2.3.4] リフレッシュトークンの実装

- [ ] 未完了
- 開始日: 未定

## 概要

Auth0から取得したリフレッシュトークンを使用して、アクセストークンの自動更新機能を実装する。JWTアクセストークンの有効期限が切れた際に、リフレッシュトークンを使用して新しいアクセストークンを取得し、ユーザーエクスペリエンスを向上させる。

## 実装内容

### 1. Auth0Service の拡張

#### 1.1 リフレッシュトークン取得処理
- `exchangeCodeForTokens` メソッドでリフレッシュトークンも取得するよう拡張
- Auth0のトークンエンドポイントから `refresh_token` を取得
- `scope` パラメータに `offline_access` を含める設定

#### 1.2 トークン更新処理
- `refreshAccessToken` メソッドの実装
- リフレッシュトークンを使用してアクセストークンを更新
- 新しいアクセストークンとリフレッシュトークンの取得
- エラーハンドリング（無効なリフレッシュトークン、期限切れなど）

### 2. SessionService の拡張

#### 2.1 リフレッシュトークン保存
- セッションデータにリフレッシュトークンを保存
- `sessionData.refreshToken` フィールドの追加
- セキュアな保存（暗号化推奨）

#### 2.2 自動トークン更新
- アクセストークンの有効期限チェック
- 期限切れ前の自動更新処理
- セッションの延長処理

### 3. 認証ミドルウェアの拡張

#### 3.1 トークン有効期限チェック
- JWTトークンの `exp` クレーム検証
- 期限切れ検知時の自動更新処理
- 更新失敗時のエラーハンドリング

#### 3.2 レスポンスヘッダー設定
- 新しいアクセストークンのレスポンスヘッダー設定
- クライアント側での新しいトークン受信処理

### 4. APIエンドポイントの追加

#### 4.1 POST /api/v1/auth/refresh
- リフレッシュトークンを受け取り、新しいアクセストークンを返却
- 手動でのトークン更新用エンドポイント
- レート制限の実装

#### 4.2 エラーレスポンスの標準化
- 401 Unauthorized: トークンが無効または期限切れ
- 403 Forbidden: リフレッシュトークンが無効
- 429 Too Many Requests: レート制限に達した場合

### 5. セキュリティ考慮事項

#### 5.1 リフレッシュトークンの安全な管理
- リフレッシュトークンの暗号化保存
- セッションタイムアウトの設定
- リフレッシュトークンローテーション（推奨）

#### 5.2 セキュリティヘッダー
- `Set-Cookie` でのセキュア属性設定
- SameSite属性の適切な設定
- HTTPOnly属性の設定

## メモ

### 実装時の注意点

1. **Auth0設定の確認**
   - Auth0 Applicationで `Refresh Token Rotation` の設定
   - `offline_access` スコープの有効化
   - トークンの有効期限設定

2. **エラーハンドリング**
   - ネットワークエラー、Auth0サービス障害への対応
   - リフレッシュトークンの無効化時の適切なログアウト処理
   - 並行リクエスト時の競合状態回避

3. **パフォーマンス考慮**
   - トークン更新のキャッシュ戦略
   - 不要なAuth0 API呼び出しの削減
   - セッションストレージの最適化

4. **セキュリティベストプラクティス**
   - リフレッシュトークンの適切な保存期間設定
   - トークンローテーションの実装検討
   - ログ出力時の機密情報マスキング
